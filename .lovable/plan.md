**Root cause of "orosz irodalom → Orosz Ferenc"**: the adj+noun guard adds "irodalom" to expansion, but the RPC still rewards `people @> '{Orosz}'` array matches and `people` text matches with no penalty when intent is `topic`. Person entity boost is unconditional.

### Scope discipline

Classifier is at 3% — making search **require** classification data would degrade quality everywhere. Treat classification as an *opportunistic boost*, not a hard signal, until coverage > 60%.

No changes to: Smart Player, sitemap/robots, classifier drain, autocomplete logic beyond what's needed for "after → Hold After Hours".

### Implementation (5 focused changes)

**1. Person-vs-topic disambiguation in `search-hybrid` (frontend of the ranking)**

- Pass new flags into `search_episodes_hybrid` RPC: `p_topic_intent boolean`, `p_demote_surname text` (the adjective when adj+noun guard fires, e.g. `"orosz"`), `p_intent text`.
- When `topic_intent=true`: do not promote person entities to `requiredTermsBase`; skip person `entities` array boost; if `p_demote_surname` matches an episode's `people[]` *without* matching a topic/category term, apply −0.4 to final score.
- When `intent='person'` AND entity resolves to a `people` row with high similarity: keep current person boost.
- Adj+noun guard already supplies the adjective; surface it explicitly in the call.

**2. RPC `search_episodes_hybrid` — add classification + relevance-review signals (opportunistic)**

- Migration: extend RPC to LEFT JOIN `episode_ai_classifications eac` and `episode_topic_relevance_reviews etrr`.
- New score contributions (only when row exists):
  - +0.15 if `eac.primary_category` or `eac.secondary_categories` matches `understanding.expanded_terms` / category-slug map
  - +0.10 per `eac.topics[]` match
  - −0.30 if `eac.rejected_topics[]` includes a query term (strong negative)
  - +0.10 if `etrr.status='accepted'` for query-matched topic
  - −0.25 if `etrr.status='rejected'` for query-matched topic
- All weighted by `eac.confidence`. NULL rows = neutral (no penalty). Falls back to current behavior on coverage gaps.

**3. Podcast-title intent**

- After `resolve_query_entities`, if top resolved entity is `kind='podcast'` AND similarity ≥ 0.85 AND query normalized matches podcast title prefix or full title: set `podcast_title_intent=true`, return that podcast's last 5 episodes (ordered `published_at DESC`) as the **top block**, semantic results below.
- Add new field `podcast_card` to response (id, title, slug, image) so the SearchPage can render a header card.
- "after" autocomplete: extend `search-autocomplete` to also match podcast titles via trigram on prefix tokens, not just exact start.

**4. Frontend (SearchPage)**

- Render `podcast_card` block + "Legújabb epizódok ebből a podcastból" when present.
- No other UI changes.

**5. Golden test harness**

- New script `scripts/search-golden.mjs` running the 25 queries through `search-hybrid` and printing intent + top-10 + classification-coverage badge. Snapshot before/after for the audit report. Not wired to CI (one-off).

### Performance

All new contributions are LEFT JOIN + index lookup (`episode_ai_classifications.episode_id` PK). Expected delta: +5–15ms p95. Embedding/Cohere paths unchanged. No new AI calls per request.

### Acceptance

- `orosz irodalom` / `orosz kultúra` top-10: zero episodes whose only signal is `people @> '{Orosz Ferenc}'`.
- `Orosz Ferenc` still returns Orosz Ferenc episodes (intent=person path preserved).
- `Hold After Hours` returns a podcast card + latest 5 HAH episodes first.
- `after` autocomplete suggests Hold After Hours in top 3.
- Golden set: ≥18/25 improved or unchanged, 0 worsened.
- p95 latency within +20ms of current.
